P-01  [HIGH]  Missing .env.example file                              [FIXED]
      The client uses two environment variables that have no documentation:
        VITE_HUD_API_URL         (default: http://localhost:3000/api/v1)
        VITE_TRANSCRIPT_WS_URL   (default: ws://localhost:3000/ws/transcript)
      Without an .env.example, new developers have no reference for local
      setup or production deployment. Production builds silently fall back to
      localhost URLs, making misconfiguration invisible.
      Fix: created client/.env.example with both vars, dev defaults, and
      commented-out production examples.

P-02  [MEDIUM]  VITE_* variables not declared in vite-env.d.ts          [FIXED]
      The file contained only `/// <reference types="vite/client" />`.
      TypeScript infers all import.meta.env.VITE_* reads as `string |
      undefined` but does not validate variable names or catch typos.
      A misspelled variable name (e.g. VITE_HUD_API_URl vs VITE_HUD_API_URL)
      compiles without error and silently uses the fallback.
      Fix: extended ImportMetaEnv in vite-env.d.ts with readonly string fields
      for both VITE_HUD_API_URL and VITE_TRANSCRIPT_WS_URL; also added
      ImportMeta interface augmentation so TS infers the env shape precisely.

P-03  [HIGH]  No GitHub Actions workflow for the client                [FIXED]
      The server has .github/workflows/server-ci.yml (typecheck, test, build,

  RESILIENCE / ARCHITECTURE
────────────────────────────────────────────────────────────────────────────────

P-04  [HIGH]  No React Error Boundary                                 [FIXED]
      Any unhandled JavaScript error thrown during a component render crashed
      the entire app tree silently (blank page with no user feedback). This was
      especially risky in TranscriptPanel (high render frequency) and App.tsx
      (complex state interactions).
      Fix: created src/app/ErrorBoundary.tsx (class component, getDerived-
      StateFromError + componentDidCatch with console.error). Wired into
      routes.tsx wrapping the ProtectedRoute + DashboardLayout + App subtree.
      Renders a MUI-styled "Reload page" fallback when a render error occurs.

P-05  [HIGH]  WebSocket connection carries no authentication token       [FIXED]
      createTranscriptSocket connected to VITE_TRANSCRIPT_WS_URL with no
      credentials, making authenticated WS sessions impossible.
      Fix: added optional token? to TranscriptSocketOptions; when non-null,
      the token is appended as ?token=<value> via URL.searchParams (no
      manual string concatenation). useTranscriptStream accepts accessToken?,
      stores it in a ref (so token rotation doesn't force a mid-session
      reconnect), and passes it to createTranscriptSocket on each connect/
      reconnect. DESKTOP_SENTINEL is filtered out before passing. App.tsx
      supplies accessToken from useAuth(). Three new socket URL tests added.

P-06  [MEDIUM]  httpOnly cookie migration (long-term security goal)      [FIXED]
      Refresh token was stored in sessionStorage, which is accessible to
      JavaScript and vulnerable to XSS. The production standard is an
      httpOnly, Secure, SameSite=Strict cookie — immune to script access.
      Fix (client side, server coordination required to deploy):
        - auth/types.ts: removed refreshToken from AuthState; TokenPair.
          refreshToken is now optional (server may still return it in JSON,
          client ignores it).
        - authApi.ts: added credentials: "include" to all auth requests so
          the browser sends/receives the httpOnly cookie automatically.
          apiRefresh() and apiLogout() now take no arguments — the cookie
          identifies the session server-side.
        - useAuthStore.tsx: removed all sessionStorage reads/writes. Mount
          effect always calls apiRefresh() (cookie sent automatically). No
          bail-out on absent stored token — the server 401s if cookie is
          missing. persistUser() stores display info in localStorage only.
        - useAuthStore.test.tsx: all tests updated for cookie-based model;
          "no-token bail-out" test replaced with "apiRefresh always called
          on mount" test; logout assertion changed to toHaveBeenCalledWith().

P-07  [MEDIUM]  Joining a new session does not clear existing transcript state [FIXED]
      When the user changed the Session ID field and clicked "Join", the
      transcript items, signals, and prompts from the previous session remained
      visible until the WebSocket delivered the new session:state message.
      Fix: added a dedicated useEffect([sessionId]) in useTranscriptStream that
      clears items/prompts/signals/errorMessage via startTransition on the
      leading edge of a session change, before the WS connects and before IDB
      restore runs. Previous session data is now never visible for a new join.
